## Explaining Semantic Search With an Incident Table in Oracle 26ai

### Introduction: From Keyword Search to Meaning-Aware Systems

Enterprise systems are overflowing with text: incident tickets, support emails, KYC documents, audit notes, product requirements, and chat transcripts. Most of this information is still accessed using **keyword search** and rigid filters.

That worked when data volumes were moderate and vocabulary was predictable. It breaks down when:

- Different teams describe the **same issue with different words**  
- You need to search **across** logs, tickets, runbooks, and documentation  
- You want applications and copilots to “understand” text, not just match strings  

This is where **semantic search** becomes essential. Instead of matching exact words, semantic search matches **meaning**, using machine learning embeddings and vector similarity.

With **Oracle Autonomous Database 26ai**, you don’t need a separate vector database to achieve this. You can:

- Store embeddings in native `VECTOR` columns  
- Generate them in-database via `DBMS_VECTOR.UTL_TO_EMBEDDING`  
- Index them using HNSW or IVF vector indexes  
- Query them with SQL using `VECTOR_DISTANCE()`  
- Orchestrate LLMs with Select AI for natural-language RAG-style workflows  

In this article, we’ll:

- Build a **semantic incident search** feature over an `incident_info` table  
- Explain why this pattern generalizes to many enterprise use cases  
- Show how this is effectively **RAG inside Oracle Autonomous Database**  

---

### What Is Semantic Search (and Why It Matters)

**Semantic search** finds results based on **meaning and intent**, not only on exact keywords.

- Keyword search: “account locked” only finds rows that literally contain that phrase.  
- Semantic search: understands that “too many failed logins” or “authentication blocked after retries” may describe the **same class of problem**.

Under the hood:

- Text is converted into a **vector embedding** (a numeric fingerprint of meaning).  
- Queries are also converted into embeddings.  
- The system returns rows whose vectors are **closest** to the query vector, according to a distance metric like **cosine** or **Euclidean**.  

For operations, support, and enterprise apps, this means:

- Faster **pattern recognition** (what does this incident look like?).  
- Better **knowledge reuse** (how did we fix similar issues?).  
- More intelligent **copilots** and assistants that can work directly over existing data.

### The Core Example: Semantic Search Over an `incident_info` Table

We’ll use a simple but realistic schema:

- `service_name`: which system is impacted (Payments, Identity, Database, Network, Trading, etc.)  
- `severity`: P1, P2, P3  
- `problem_text`: human-written incident description  
- `resolution_text`: how we resolved it  
- `problem_vec`, `resolution_vec`: vector embeddings of the problem and resolution text  

The goal:

- Given a **new incident description**, find **similar historical incidents** and show their resolutions.  
- Optionally, check whether a **proposed resolution** is similar to past successful resolutions.  

All of this happens **inside Oracle Autonomous Database** using SQL.

---

### Step 1 – Create the Incident Table With VECTOR Columns

```sql
BEGIN
  EXECUTE IMMEDIATE 'DROP TABLE incident_info PURGE';
EXCEPTION WHEN OTHERS THEN NULL;
END;
/

CREATE TABLE incident_info (
  incident_id      NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  service_name     VARCHAR2(100),
  severity         VARCHAR2(20),
  created_at       TIMESTAMP DEFAULT SYSTIMESTAMP,
  problem_text     CLOB,
  resolution_text  CLOB,
  problem_vec      VECTOR(1536, FLOAT32),
  resolution_vec   VECTOR(1536, FLOAT32)
);
```

- `VECTOR(1536, FLOAT32)` matches the output dimension of OpenAI’s `text-embedding-3-small` model.  
- We keep both the **raw text** and its **meaning representation** in a single row.

---

### Step 2 – Insert 200 Realistic Incidents

Instead of hand-writing rows, we use a PL/SQL loop:

```sql
BEGIN
  FOR i IN 1..200 LOOP
    INSERT INTO incident_info (
      service_name,
      severity,
      problem_text,
      resolution_text
    )
    VALUES (
      CASE MOD(i,5)
        WHEN 0 THEN 'Payments'
        WHEN 1 THEN 'Identity'
        WHEN 2 THEN 'Database'
        WHEN 3 THEN 'Network'
        ELSE        'Trading'
      END,
      CASE
        WHEN MOD(i,20) = 0 THEN 'P1'
        WHEN MOD(i,5)  = 0 THEN 'P2'
        ELSE                 'P3'
      END,
      'INC-'||LPAD(i,4,'0')||': '||
      CASE MOD(i,6)
        WHEN 0 THEN 'Customer reports timeouts during checkout after a firewall policy update.'
        WHEN 1 THEN 'Login failures increased; some users see account locked messages.'
        WHEN 2 THEN 'Batch job is slow; CPU utilization is high and queries are waiting on I/O.'
        WHEN 3 THEN 'Intermittent connection resets observed between app tier and database.'
        WHEN 4 THEN 'Unexpected latency spike after deployment; error rate elevated for one API.'
        ELSE        'Data load failed with invalid format; downstream reports are stale.'
      END || ' Impact: ' ||
      CASE MOD(i,4)
        WHEN 0 THEN 'high'
        WHEN 1 THEN 'medium'
        WHEN 2 THEN 'low'
        ELSE        'moderate'
      END,
      'Resolution for INC-'||LPAD(i,4,'0')||': '||
      CASE MOD(i,6)
        WHEN 0 THEN 'Allow outbound HTTPS to required endpoints, update ACL/firewall rules, restart affected pods.'
        WHEN 1 THEN 'Block abusive IPs, adjust lockout threshold, reset impacted accounts, confirm MFA policy.'
        WHEN 2 THEN 'Add missing indexes, gather optimizer stats, tune top SQL, scale resources during peak.'
        WHEN 3 THEN 'Fix load balancer idle timeout, enable keepalives, increase pool limits, validate TLS settings.'
        WHEN 4 THEN 'Rollback or hotfix deployment, review logs, add circuit breaker, and re-run health checks.'
        ELSE        'Fix input validation, correct file format mapping, re-run pipeline, and backfill reports.'
      END
    );
  END LOOP;
  COMMIT;
END;
/
```

Now we have a small but rich dataset spanning multiple services and severities.

### Step 3 – Configure DBMS_VECTOR to Use an Embedding Provider

`DBMS_VECTOR.UTL_TO_EMBEDDING` can call:

- In-database ONNX models (`provider: "database"`), or  
- Third-party REST providers like OpenAI, Cohere, Google AI, etc.  

For OpenAI, we create a credential and allow outbound HTTPS:

```sql
BEGIN
  DBMS_VECTOR.DROP_CREDENTIAL('OPENAI_CRED_DEMO');
EXCEPTION WHEN OTHERS THEN NULL;
END;
/

DECLARE
  jo json_object_t;
BEGIN
  jo := json_object_t();
  jo.put('access_token', '<YOUR_OPENAI_KEY>');

  DBMS_VECTOR.CREATE_credential(
    credential_name => 'OPENAI_CRED_DEMO',
    params          => JSON(jo.to_string)
  );
END;
/
```

```sql
BEGIN
  DBMS_NETWORK_ACL_ADMIN.APPEND_HOST_ACE(
    host => 'api.openai.com',
    ace  => xs$ace_type(
             privilege_list => xs$name_list('connect'),
             principal_name => '<YOUR_DB_USER>',
             principal_type => xs_acl.ptype_db
           )
  );
END;
/
```

> Note: in a production system you’d likely use OCI Generative AI (`provider: "ocigenai"`) or in-database ONNX models. The pattern is the same.

### Step 4 – Generate Problem and Resolution Embeddings in-DB

We now call `UTL_TO_EMBEDDING` for each row:

```sql
DECLARE 
  params CLOB;
BEGIN
  params := '{
    "provider": "openai",
    "url": "https://api.openai.com/v1/embeddings",
    "credential_name": "OPENAI_CRED_DEMO",
    "model": "text-embedding-3-small"
  }';

  UPDATE incident_info i
     SET i.problem_vec =
           DBMS_VECTOR.UTL_TO_EMBEDDING(i.problem_text, JSON(params)),
         i.resolution_vec =
           DBMS_VECTOR.UTL_TO_EMBEDDING(i.resolution_text, JSON(params))
   WHERE i.problem_text IS NOT NULL
     AND i.resolution_text IS NOT NULL;

  COMMIT;
END;
/
```

Every incident now has:

- `problem_vec`: numeric representation of the problem description  
- `resolution_vec`: numeric representation of the resolution summary  

These are first-class citizens in Oracle via the `VECTOR` type.

---

### Step 5 – Create HNSW Vector Indexes for Fast Semantic Search

Exact vector search (full scan) works when you have a few hundred rows. At larger scale, you’ll want **approximate nearest neighbor (ANN)** indexes.

Oracle supports HNSW (Hierarchical Navigable Small World) via:

```sql
BEGIN
  EXECUTE IMMEDIATE 'DROP INDEX inc_problem_hnsw_idx';
EXCEPTION WHEN OTHERS THEN NULL;
END;
/
BEGIN
  EXECUTE IMMEDIATE 'DROP INDEX inc_resolution_hnsw_idx';
EXCEPTION WHEN OTHERS THEN NULL;
END;
/

CREATE VECTOR INDEX inc_problem_hnsw_idx
ON incident_info (problem_vec)
ORGANIZATION INMEMORY NEIGHBOR GRAPH
DISTANCE COSINE
WITH TARGET ACCURACY 95;

CREATE VECTOR INDEX inc_resolution_hnsw_idx
ON incident_info (resolution_vec)
ORGANIZATION INMEMORY NEIGHBOR GRAPH
DISTANCE COSINE
WITH TARGET ACCURACY 95;
```

What this does:

- **`ORGANIZATION INMEMORY NEIGHBOR GRAPH`**: build an in-memory neighbor graph (HNSW).  
- **`DISTANCE COSINE`**: use cosine distance as the similarity metric.  
- **`TARGET ACCURACY 95`**: trade a small amount of accuracy for much better latency.  

At query time, Oracle walks the graph instead of scanning all vectors, which is critical when your incident dataset grows into the millions.

### Step 6 – Query A: “Show Me Similar Incidents”

First semantic search scenario:

> Given a new incident description, find the most similar historical incidents and show their resolutions.

```sql
WITH q AS (
  SELECT DBMS_VECTOR.UTL_TO_EMBEDDING(
           'Payments failing with timeout errors',
           JSON('{
             "provider": "openai",
             "url": "https://api.openai.com/v1/embeddings",
             "credential_name": "OPENAI_CRED_DEMO",
             "model": "text-embedding-3-small"
           }')
         ) AS qvec
  FROM dual
)
SELECT i.incident_id,
       i.service_name,
       i.severity,
       VECTOR_DISTANCE(i.problem_vec, q.qvec, COSINE) AS prob_dist,
       i.problem_text,
       i.resolution_text
FROM   incident_info i
CROSS  JOIN q
WHERE  i.problem_vec IS NOT NULL
ORDER  BY prob_dist
FETCH APPROXIMATE FIRST 5 ROWS ONLY;
```

Here:

- We embed the **query text** once as `qvec`.  
- `VECTOR_DISTANCE(problem_vec, qvec, COSINE)` measures semantic distance.  
- Smaller distance = more similar incident.  
- `FETCH APPROXIMATE` tells Oracle to use the HNSW index.  

In practice, this retrieves incidents like:

- Other payment timeouts  
- Firewall or network-related issues  
- Similar customer impact scenarios  

All through a single SQL query.

### Step 7 – Query B: Evaluating Problem and Resolution Fit

Sometimes you don’t just want “similar problems”; you also care whether the **way you solved it** is similar.

We can embed both:

- A **problem query**: what this incident looks like  
- A **resolution query**: how we think it should be fixed  

Then combine distances:

```sql
WITH q AS (
  SELECT
    DBMS_VECTOR.UTL_TO_EMBEDDING(
      'Payments failing with timeout errors after recent firewall change',
      JSON('{"provider":"openai","url": "https://api.openai.com/v1/embeddings","credential_name":"OPENAI_CRED_DEMO","model":"text-embedding-3-small"}')
    ) AS prob_q,
    DBMS_VECTOR.UTL_TO_EMBEDDING(
      'Update firewall/ACL to allow outbound TLS to payment gateway and restart services',
      JSON('{"provider":"openai","url": "https://api.openai.com/v1/embeddings","credential_name":"OPENAI_CRED_DEMO","model":"text-embedding-3-small"}')
    ) AS res_q
  FROM dual
)
SELECT i.incident_id,
       i.service_name,
       i.severity,
       VECTOR_DISTANCE(i.problem_vec,    q.prob_q, COSINE) AS prob_dist,
       VECTOR_DISTANCE(i.resolution_vec, q.res_q,  COSINE) AS res_dist,
       (VECTOR_DISTANCE(i.problem_vec, q.prob_q, COSINE) * 0.7
        + VECTOR_DISTANCE(i.resolution_vec, q.res_q, COSINE) * 0.3) AS combined_score,
       i.problem_text,
       i.resolution_text
FROM   incident_info i
CROSS  JOIN q
WHERE  i.problem_vec IS NOT NULL
  AND  i.resolution_vec IS NOT NULL
ORDER  BY combined_score
FETCH APPROXIMATE FIRST 5 ROWS ONLY;
```

This gives you:

- `prob_dist`: how similar the problem is  
- `res_dist`: how similar the resolution is  
- `combined_score`: weighted combination (70% problem, 30% resolution)  

You can tune these weights per use case.

### How This Becomes RAG Inside Oracle Autonomous Database

**RAG (Retrieval-Augmented Generation)** has two core steps:

1. **Retrieve** relevant context (documents, incidents, knowledge) for a query.  
2. **Augment generation** by feeding that context into an LLM to produce an answer, explanation, or plan.  

With Oracle Autonomous Database 26ai:

- **Retrieval** is done by:
  - Storing embeddings in `VECTOR` columns.  
  - Using `VECTOR_DISTANCE()` with HNSW/IVF indexes to get the top‑K rows.  
- **Augmentation + Generation** can be handled by:
  - **Select AI**: natural-language-to-SQL over both structured and vectorized data.  
  - `DBMS_CLOUD_AI.GENERATE` or application-level calls to LLMs, passing the retrieved rows as context.  

Concrete pattern:

1. User asks: “Why are payments timing out after last night’s deployment?”  
2. Database:
   - Uses `UTL_TO_EMBEDDING` + `VECTOR_DISTANCE()` to find **similar incidents, logs, and runbooks**.  
   - Optionally filters by `service_name = 'Payments'` and `created_at >= SYSDATE - 365`.  
3. An LLM (invoked via Select AI or application code) receives:
   - The user question.  
   - The retrieved incidents and resolutions as context.  
4. LLM generates:
   - A summarized explanation.  
   - Recommended next steps (“check firewall ACLs, confirm payment gateway connectivity, restart pods”).  

All of this uses Oracle as the **retrieval engine and primary data store**, avoiding external vector DBs and keeping governance centralized.

### Beyond Incidents: Other Enterprise Use Cases

The same pattern applies far beyond incident management:

- **Customer Support & CRM**
  - Vectorize support tickets, chat transcripts, and knowledge base articles.  
  - Build “similar case” search and automated answer suggestion.  

- **Financial Services (AML / KYC / Credit)**
  - Semantic search over case narratives, SARs, and investigator notes.  
  - Cross-link similar typologies (e.g., mule accounts, layering, structuring).  

- **Legal & Compliance**
  - Semantic search over contracts, policies, and regulatory texts.  
  - RAG applications that generate summaries and impact analyses for new regulations.  

- **Product & Engineering Knowledge**
  - Search across design docs, RFCs, code review discussions, and runbooks.  
  - Help new engineers and SREs find “how we solved this before” in seconds.  

In every case, the ingredients are the same:

- Store business objects + documents in Oracle.  
- Add `VECTOR` columns to hold their embeddings.  
- Use `DBMS_VECTOR` to generate embeddings (database or REST providers).  
- Index with HNSW/IVF.  
- Query with SQL and feed the results to LLMs or apps.

Oracle Autonomous Database 26ai brings this capability **into the database itself**:

- `VECTOR` columns and `DBMS_VECTOR.UTL_TO_EMBEDDING` for storing meaning, not just strings  
- HNSW and IVF vector indexes for fast similarity search  
- `VECTOR_DISTANCE()` and standard SQL for retrieval logic  
- Select AI and database-side LLM integrations for RAG-style workflows  

The incident example in this article is just one concrete pattern. The same tools can power semantic search and RAG across almost any enterprise dataset already sitting in Oracle.
